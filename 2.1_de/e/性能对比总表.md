## 🎯 结论总结

### 三个关键发现

#### ✅ 1. ijkijk 平时性能优秀
- 加速比：**2.10x ~ 2.25x**（稳定）
- 相对性能：**2.1x 更快**

#### ⚠️ 2. 1024 是缓存陷阱
- **ijk**：小幅下降（+7.5%）
- **ijkijk**：严重下降（+94.5%）⚠️
- 原因：2^n 规模导致 Cache 组冲突

#### ✅ 3. 即使陷入陷阱，ijkijk 仍优于 ijk
```
1024 规模下：
  ijk：834.89ms
  ijkijk：739.78ms
  ijkijk 仍快 12.8%
```

## 💡 优化建议

### 第一优先级：使用分块乘法
```c
// ✓ 推荐：ijkijk 分块
for (int ii = 0; ii < n; ii += BLOCK_SIZE)
    for (int jj = 0; jj < n; jj += BLOCK_SIZE)
        for (int kk = 0; kk < n; kk += BLOCK_SIZE)
            // 6 层嵌套
```

### 第二优先级：避免 2^n 矩阵规模
```c
// ✓ 推荐
int n = 1001;   // 非 2^n
int n = 1023;   // 2^n - 1
int n = 1025;   // 2^n + 1

// ✗ 避免
int n = 1024;   // 2^10 陷阱
int n = 2048;   // 2^11 陷阱
```

### 第三优先级：必要时使用 Padding
```c
// 如果必须用 1024，可以用 Padding 破坏对齐
double A[1024][1032];  // +8 列填充
// 这样可以恢复到 ~2.0x 的加速比
```
