## 📄 第 10 页：原因分析

### 标题
**缓存冲突陷阱：为什么 1024 这么特殊？**

---

### 内容结构

#### 上半部分：根本原因（简化版）

```
【Cache 组冲突机制】

L2 Cache 结构：256KB = 512组 × 8路 × 64B行

地址映射规则：
  组索引 = (物理地址 >> 6) & 0x1FF

问题：当矩阵规模为 1024 时
  每行占 1024×8 = 8192 字节 = 2^13 字节
  
  A[0] 起始 → 组 0
  A[1] 起始 → 组 128
  A[2] 起始 → 组 256
  A[3] 起始 → 组 384
  A[4] 起始 → 组 0  ← 和 A[0] 冲突！❌
```

#### 下半部分：为什么分块被击中更硬

```
【访问模式对比】

ijk 直接乘法：
  • 访问模式散乱，缓存冲突已存在
  • 1024 处冲突增加相对较小 (+7.5%)

ijkijk 分块乘法：
  • 块内访问非常规则
  • 块之间地址对齐，冲突集中爆发
  • 性能下降严重 (+94.5%) ⚠️

结论：规则的访问模式对缓存冲突更敏感

【解决方案】
✓ 使用分块乘法（ijkijk） → 平时快 2.1 倍
✓ 避免 2^n 规模（1023 或 1025）
✓ 必要时使用 Padding 破坏对齐
```

## 🎯 精简写法示例

### ❌ 不要这样写（太长）
```
当矩阵规模为 1024×1024 时，由于 1024 是 2 的 10 次方，
矩阵每行占用的字节数恰好是 1024 乘以 8 等于 8192 字节，
这个数值也是 2 的 13 次方。在 CPU 的 L2 Cache 中，
采用了组相联结构，每组有 512 个位置...
```

### ✅ 这样写（精简有力）
```
问题：1024 = 2^10
      每行 8192 = 2^13 字节
      
结果：多行映射到同一 Cache 组
      频繁驱逐 → 命中率崩溃
      
性能：ijkijk 从 5.6G 跌至 2.9G (-49%)
```
