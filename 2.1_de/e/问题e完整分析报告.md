# 问题 e：矩阵规模对缓存性能的影响

## 一、实验概述

### 研究问题
当矩阵规模为 2 的幂次（如 1024×1024）时，为什么性能会出现陡峭下降？

### 实验设计
对比两种算法在不同矩阵规模下的性能：
1. **ijk 直接乘法**：标准的三层嵌套循环，无优化
2. **ijkijk 分块乘法**：块大小 64×64，6 层嵌套循环（块级 ijk + 元素级 ijk）

### 测试规模
重点关注 1024 附近的规模变化：
- **1001～1023**：非 2 的幂，基准
- **1024**：2^10，关键观察点  
- **1025**：2^10 + 1，对比
- **1030～1500**：后续规模观察

## 三、关键发现

### 3.1 1024 处的性能崩溃现象

#### ijk 直接乘法
- 1023 → 1024：耗时增加 **7.5%**（776.85ms → 834.89ms）
- GFLOPS 下降：2.7563 → 2.5722
- **性能下降不明显，但能看到跳变**

#### ijkijk 分块乘法  
- 1023 → 1024：耗时增加 **94.5%**（380.22ms → 739.78ms）
- GFLOPS 下降：5.6315 → 2.9029（**下降 49%**）
- **性能严重崩溃！加速比从 2.04x 跌至 1.13x**

**结论**：分块乘法对缓存冲突更敏感，在 1024 处性能灾难级下降。

### 3.2 为什么 1024 这么特殊？

#### 缓存冲突机制

在 Apple Silicon 上，L2 Cache 结构：
- 大小：256 KB
- 组数：512 组（0～511）
- 路数：8 路（8-way associative）
- 行大小：64 字节

#### 地址映射规则

```
组索引 = (物理地址 >> 6) & 0x1FF    // 取地址的 bit 6～14
```

#### 为什么 1024×1024 会冲突

当矩阵规模为 1024 时：
```
每行占用字节数 = 1024 × 8 = 8192 = 2^13 字节

A[0][0] 起始地址：0x0
A[1][0] 起始地址：0x2000      (偏移 8192 = 0x2000)
A[2][0] 起始地址：0x4000      (偏移 16384)
...
A[256][0] 起始地址：0x200000   (偏移 2097152)

组索引计算：
A[0][0]:   (0x0 >> 6) & 0x1FF = 0x00
A[1][0]:   (0x2000 >> 6) & 0x1FF = 0x80
A[2][0]:   (0x4000 >> 6) & 0x1FF = 0x100
A[3][0]:   (0x6000 >> 6) & 0x1FF = 0x180
A[4][0]:   (0x8000 >> 6) & 0x1FF = 0x00   ← 和 A[0][0] 冲突！
```

**规律**：每隔 4 行（4 × 8192 = 32768 = 0x8000），两行会映射到同一个组。

这导致：
- 多行竞争同一个缓存组的 8 个位置
- 频繁的驱逐和重新加载
- **缓存命中率急剧下降**

#### 为什么 1025 没有问题

```
每行占用字节数 = 1025 × 8 = 8200 字节

8200 = 0x2008   (不是 2^n 对齐)

组索引分布更均匀，不会产生集中冲突
```

## 四、深度分析

### 4.1 为什么分块乘法被击中更硬

对比 ijk 和 ijkijk：

| 算法 | L1 Cache 局部性 | 对冲突的敏感度 |
|-----|-------------|------------|
| **ijk 直接** | 较差 | 中等 |
| **ijkijk 分块** | 很好（平时） | **极高** |

**原因**：
- ijk 直接乘法：访问模式较散乱，缓存冲突已经存在，1024 处的额外冲突影响相对较小
- ijkijk 分块乘法：在块内访问非常规则，块与块之间的地址对齐使得冲突集中爆发

### 4.2 恢复性分析

从 1024 回到 1025：
- **ijk**：从 834.89ms → 785.07ms（恢复 6.0%）
- **ijkijk**：从 739.78ms → 399.00ms（恢复 46.1%，迅速恢复！）

这说明：**只要打破 2^n 对齐，冲突就消失**

---

## 五、结论与建议

### 5.1 核心结论

1. **1024×1024 是"缓存陷阱"**
   - 2^n 对齐导致多行映射到同一缓存组
   - 分块乘法被击中最硬：性能下降 ~94%

2. **非 2^n 规模性能稳定**
   - 地址分布随机，冲突少
   - 分块乘法始终保持 2.0x～2.25x 加速

3. **分块乘法本质更优**
   - 即使在 1024 的坏情况下，ijkijk（739.78ms）仍快于 ijk（834.89ms）
   - 平时的加速比达 2.1x～2.3x

### 5.2 实际建议

#### 避免 2^n 矩阵规模
```c
// 不推荐
int n = 1024;  // 2^10

// 推荐
int n = 1001;  // 非 2^n
int n = 1023;  // 2^n - 1
int n = 1025;  // 2^n + 1
```

#### 如果必须用 2^n，使用 Padding
```c
// 原始（会冲突）
double A[1024][1024];

// 优化（破坏对齐）
double A[1024][1032];  // 多 8 列填充
```

#### 使用分块乘法（ijkijk）
- 大部分情况下性能提升 2.1x
- 即使遇到缓存陷阱，分块仍优于直接乘法

## 六、技术细节

### 缓存结构参数（Apple Silicon M-series）

```
L1 Cache:
  大小：64 KB (数据)
  组关联：8-way
  行大小：64 B
  组数：128 组

L2 Cache（每核心）:
  大小：256 KB
  组关联：8-way
  行大小：64 B
  组数：512 组
```

### 冲突模式

对于 n×n 矩阵：
- **最坏情况**：n = 2^k，多行映射到同组
- **一般情况**：n 为任意数，冲突少
- **最好情况**：n 为质数或特殊设计的数
