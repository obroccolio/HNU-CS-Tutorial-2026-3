## 🎯 核心发现

### 1024 性能灾难

| 指标 | ijk 直接乘法 | ijkijk 分块乘法 |
|-----|-----------|-------------|
| 1023 的耗时 | 776.85 ms | 380.22 ms |
| 1024 的耗时 | 834.89 ms | **739.78 ms** |
| 性能下降 | **+7.5%** | **+94.5%** ⚠️ |
| 加速比 (1023) | 2.04x | - |
| 加速比 (1024) | **1.13x** | - |

### 原因分析

**缓存冲突机制**：

当矩阵规模为 1024 时，每行占用 8192 字节（= 2^13 字节）。在 8-way 组相联的 L2 Cache 中，地址映射规律导致多行映射到同一 Cache 组，频繁驱逐和重新加载，性能崩溃。

```
A[0][0]   → Cache 组 0
A[1][0]   → Cache 组 128
...
A[4][0]   → Cache 组 0 (冲突!)
```

**为什么分块更敏感**：

分块乘法在块内的访问非常规则，块与块之间的地址对齐使得冲突集中爆发。

## 🔍 结论

### 1. 2^n 矩阵规模是"缓存陷阱"

1024 = 2^10 规模下：
- 直接 ijk 性能下降 7.5%
- 分块 ijkijk 性能下降 **94.5%**

### 2. 分块乘法平时更优

避开 2^n 陷阱的规模：
- ijkijk 的加速比稳定在 **2.0x～2.25x**
- 即使陷入 1024 陷阱，ijkijk 仍快于 ijk

### 3. 实用建议

✅ **使用分块乘法（ijkijk）**
✅ **避免 2^n 矩阵规模**（或用 2^n ± 1）
✅ **必要时使用 Padding 破坏对齐**
