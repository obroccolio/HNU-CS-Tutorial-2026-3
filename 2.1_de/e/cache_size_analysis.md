### 2️⃣ 为什么2的幂（1024、2048）这么糟糕？

#### 根本原因：Cache组冲突

现代CPU的L2/L3 cache是**组相联**（set-associative）结构：

```
Cache大小  = 组数 × 路数 × 行大小
  256KB  = 512组 × 8路 × 64B

组索引 = (物理地址 >> 6) & (512-1)  // 取bit6~14
        = (物理地址 >> 6) % 512
        = (物理地址 >> 6) & 0x1FF   // 与511按位与
```

#### 当矩阵宽度是2的幂时

```
矩阵A的行宽 = 1024 * sizeof(double) = 1024 * 8 = 8192 字节

行0 起始: 地址 0x0
行1 起始: 地址 0x2000      (8192 = 0x2000)
行2 起始: 地址 0x4000
行3 起始: 地址 0x6000
...

组索引计算（假设组数=512，行大小64B）：
行0: (0x0 >> 6) & 0x1FF = 0x000
行1: (0x2000 >> 6) & 0x1FF = (0x80) & 0x1FF = 0x80
行2: (0x4000 >> 6) & 0x1FF = (0x100) & 0x1FF = 0x100
行3: (0x6000 >> 6) & 0x1FF = (0x180) & 0x1FF = 0x180
行4: (0x8000 >> 6) & 0x1FF = (0x200) & 0x1FF = 0x000  ← 和行0冲突！
行5: (0xA000 >> 6) & 0x1FF = (0x280) & 0x1FF = 0x80   ← 和行1冲突！
```

**问题**：每隔若干行，就会映射到同一个cache组！

当多行要竞争同一组的8个位置时 → 频繁驱逐 → 缓存命中率急剧下降

#### 当矩阵宽度不是2的幂时

```
矩阵宽度 = 1025 * 8 = 8200 字节

行0: 地址 0x0
行1: 地址 0x2008
行2: 地址 0x4010
...

组索引：
行0: (...) & 0x1FF = 0x000
行1: (...) & 0x1FF = 0x010  ← 不同的组
行2: (...) & 0x1FF = 0x020  ← 不同的组
...

分布更均匀，不会在同一组产生集中冲突
```

---

## 证明：缓存冲突是罪魁祸首

### 转置后性能崩溃的理由

转置操作会：
1. 额外占用内存（需要转置后的矩阵副本）
2. 增加cache的内存压力
3. **在1024处，由于组冲突，cache利用率已经很低** → 转置副本进一步压低了it,导致加速比从0.46x变成0.24x

```
不转置(1024):
  ├─ 原始cache冲突已存在  → miss增加
  └─ 但只需A、B两个矩阵 → 相对还能容纳

转置后(1024):
  ├─ 同样的冲突！
  ├─ 额外的转置 + 转置后矩阵 → 内存占用更多
  ├─ cache压力激增
  └─ 相对指标 0.46x → 0.24x (性能崩溃70%)
```

而1025不是2的幂，避免了组冲突：
```
不转置(1025): cache利用率较好
转置后(1025): 转置开销相对可接受  → 加速比保持在0.78x
```

## 性能曲线分析

### 理论预期 vs 实际结果

**理论**（如果没有组冲突）：
- 性能曲线应该平滑上升（矩阵越大，计算越多，时间越长）
- 转置的收益应该相对一致

**实际**（考虑组冲突）：
- **平滑** 部分（1001, 1002, 1003...）：性能缓慢增长
- **崩溃** 部分（1024, 2048）：转置后性能突然下降70%
- **恢复** 部分（1025, 2049）：性能恢复正常

这就是**组相联cache的经典陷阱**！

## 为什么会这样设计？

### Cache为什么是2的幂？

```
Cache容量: 256KB, 512KB, 1MB, 2MB
           ↑所有都是2的幂

组数: 512组, 1024组, 2048组
      ↑所有都是2的幂

原因：二进制地址的bit提取最快
```

硬件设计者希望用**位运算**（高速）而不是**取模运算**（低速）来计算组索引。但这导致了"整齐"矩阵规模会产生冲突。

---

## 实际意义

### 优化建议

1. **避免矩阵规模为2的幂**（如果可能）
   - 用1025代替1024 → 性能提升70%
   - 用2049代替2048 → 性能提升70%

2. **使用padding破坏对齐**
   ```c
   // 而不是 int matrix[1024][1024]
   int matrix[1024][1024+1];  // 多一列，破坏cache对齐
   ```

3. **选择不同的分块大小**
   ```c
   int block_size = 64;  // 好
   int block_size = 128; // 也好（不冲突）
   // 避免block_size导致的二次对齐问题
   ```

## 图示：Cache冲突原理

```
CPU cache (8路，512组):
┌─────────────────────────────────────┐
│ 组0: [××××××××] ← 饱和（冲突！）     │
│ 组1: [  ］                           │
│ ...                                  │
│ 组128: [××××××××] ← 饱和（冲突！）  │
│ 组129: [  ]                          │
└─────────────────────────────────────┘

1024×1024矩阵：
  每隔4行就映射到同一组
  → 频繁驱逐 → miss增加

1025×1025矩阵：
  行宽=8200B，不与512组对齐
  → 分布均匀 → miss减少
```

## 总结

| 规模类型 | 性能 | 原因 |
|---------|------|------|
| 普通规模（1001, 1002...） | 正常 | cache分布均匀 |
| 2的幂（1024, 2048） | **崩溃** | 组冲突，cache利用率低 |
| 2^幂+1（1025, 2049） | 恢复 | 避开对齐，冲突消失 |

**性能曲线不是平滑的！在整"齐"规模处出现陡峭跳变，转置后性能最多下降70%。**

这是**组相联cache的经典陷阱**，在大规模矩阵运算中需要特别关注。
