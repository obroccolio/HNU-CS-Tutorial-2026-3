# Cache 组索引决定机制 - 完整说明

## 📚 知识回顾

### 1. L2 Cache 的基本结构

```
L2 Cache = 256 KB

组织方式：
  512 个组（Set）× 8 路（Way）× 64 字节（每行）
  
  └─ 512 组：Set 0, Set 1, ..., Set 511
  └─ 8 路：Way 0, Way 1, ..., Way 7（每个组内）
  └─ 64 字节：一条缓存行的大小
```

### 2. 缓存行（Cache Line）

```
定义：Cache 的最小存储单位

大小：64 字节
内容：可以存放 64 字节的连续数据
      例：8 个 double（每个 8 字节）
          或 16 个 int（每个 4 字节）

作用：当 CPU 访问某个地址的数据时，
     硬件会自动加载该地址周围的 64 字节进缓存
```

### 3. Way（路）

```
定义：同一个组内的多个存储位置

8-way 含义：每个组有 8 个存储位置
           每个位置可以存一条缓存行

例：
  组 0 可以同时存 8 条缓存行，分别在 Way 0～7
  组 1 也可以同时存 8 条缓存行，分别在 Way 0～7
  ...
  
好处：允许多个不同地址的数据映射到同一个组，
     只要它们存到不同的 Way，就不会冲突
```

### 4. 地址映射（Address Mapping）

```
CPU 的地址 → 确定存在 Cache 的哪个位置

两步过程：
  第一步：确定组号（Set Index）
  第二步：在该组内选择 Way（由替换策略决定）
  
结果：一个唯一的 Cache 位置
```

## 🔑 **组索引（Set Index）是什么决定的？**

### 核心原理

```
组索引由：【物理地址的某些位】决定

具体是哪些位？
  → 地址的第 6～14 位
  → 共 9 位（因为 2^9 = 512 组）
```

### 公式

```
组索引 = (物理地址 >> 6) & 0x1FF

拆解说明：
  >> 6      = 右移 6 位（跳过最低 6 位）
  & 0x1FF   = 与上 0x1FF（取出中间 9 位）
  
结果：一个 0～511 的数字
```

### 为什么是第 6～14 位？

```
第 0～5 位（最低 6 位）= 缓存行内的偏移
                    = 2^6 = 64 字节内的位置
                    = 不用来计算组索引
                    → 因为 64 字节是一条线的大小

第 6～14 位（中间 9 位）= 组号
                      = 2^9 = 512 个组
                      = 用来确定 Set Index
                      
第 15 位以上 = 标签（Tag）
            = 用来确定是否缓存命中
            = 与 Set Index 一起确定位置
```

### 用图表示

```
物理地址二进制表示：

┌─────────────────────────────────┐
│ ... │ Tag(15~) │ Set(6~14) │ Offset(0~5) │
│     │          │ (9位)    │ (6位)     │
└─────────────────────────────────┘
      └─ 确定组号
```

## 📊 **1024 和 1025 的组索引对比**

### 1024×1024 矩阵

```
每行占用：1024 × 8 = 8192 字节

行的起始物理地址：
  A[0]: 0x0
  A[1]: 0x2000 (8192)
  A[2]: 0x4000 (16384)
  A[3]: 0x6000 (24576)
  A[4]: 0x8000 (32768) ← 注意：32768 = 2^15
  A[5]: 0xA000 (40960)
  ...

组索引计算（使用公式 (地址 >> 6) & 0x1FF）：
  A[0]: (0x0 >> 6) & 0x1FF = 0
  A[1]: (0x2000 >> 6) & 0x1FF = 128
  A[2]: (0x4000 >> 6) & 0x1FF = 256
  A[3]: (0x6000 >> 6) & 0x1FF = 384
  A[4]: (0x8000 >> 6) & 0x1FF = (512) & 511 = 0 ← 回到 0！
  A[5]: (0xA000 >> 6) & 0x1FF = (640) & 511 = 128 ← 也回到 128！

规律：每隔 4 行重复
      → 组 0, 128, 256, 384, 0, 128, 256, 384, ...
      → 组 0 面临频繁的 A[0], A[4], A[8], ... 映射
      → 只有 8 个 Way，容纳不了 → 冲突！
```

### 1025×1025 矩阵

```
每行占用：1025 × 8 = 8200 字节（非 2 的幂！）

行的起始物理地址：
  A[0]: 0x0
  A[1]: 0x2008 (8200)
  A[2]: 0x4010 (16400)
  A[3]: 0x6018 (24600)
  A[4]: 0x8020 (32800) ← 注意：32800 不是 2^n
  ...

组索引计算：
  A[0]: (0x0 >> 6) & 0x1FF = 0
  A[1]: (0x2008 >> 6) & 0x1FF = 128
  A[2]: (0x4010 >> 6) & 0x1FF = 256
  A[3]: (0x6018 >> 6) & 0x1FF = 385 ← 不同！
  A[4]: (0x8020 >> 6) & 0x1FF = 1   ← 又不同！
  A[5]: (0xA028 >> 6) & 0x1FF = 129 ← 还是不同！
  ...

规律：完全分散，没有周期性
      → 很多不同的组承载 A[i]
      → 每个组的竞争很少
      → 8 个 Way 足够 → 无冲突！
```

## 💡 **核心理解**

### 组索引为什么重要？

```
组索引决定了数据在 Cache 中"进入哪个大门"
  ↓
如果多条数据的组索引相同
  ↓
它们就会竞争同一个组的 8 个 Way
  ↓
当超过 8 条时，就会发生冲突

问题：
  组索引由地址的第 6～14 位决定
  而地址增长是连续的（每次 +8200 或 +8192）
  
  如果这个增长是 2 的幂（如 8192 = 2^13）
  → 组索引会按照 2 的幂循环
  → 周期很短（每 4 行循环）
  → 冲突频繁
  
  如果这个增长不是 2 的幂（如 8200）
  → 组索引会分散
  → 周期很长（每 4096 行循环）
  → 冲突少
```

### 根本原因

```
【2^n 灾难的本质】

组索引 = (地址 >> 6) & 0x1FF
  ↑
  用的是二进制位操作（>> 和 &）
  这种操作天然对 2 的幂敏感

当矩阵行大小恰好是 2^13 时：
  每行地址增长 = 2^13
  每隔 2^13 / 2^9 = 2^4 = 16... 不对
  
让我重新算：
  Cache 组扫完一遍 = 512 × 64 = 2^15
  矩阵行大小 = 2^13
  2^15 / 2^13 = 2^2 = 4
  
  → 每 4 行回到同一组
  
当矩阵行大小是 8200 时：
  不是 2 的幂，含奇数因子
  要循环回组 0 需要很多行
  → 冲突少
```

## 📝 **总结**

### 组索引的决定方式

```
1. CPU 计算数据的物理地址
2. 提取地址的第 6～14 位
3. 这 9 位就是组索引（0～511）
4. 数据存入对应的组

组索引 = (物理地址 >> 6) & 0x1FF
```

### 为什么 1024 冲突、1025 不冲突

```
1024×1024：
  矩阵每行 = 8192 = 2^13（2 的幂）
  ↓
  地址增长也是 2 的幂
  ↓
  组索引计算（位操作）快速循环
  ↓
  每 4 行回到同一组
  ↓
  冲突频繁（8 个 Way 不够用）

1025×1025：
  矩阵每行 = 8200 = 8×1025（不是 2 的幂）
  ↓
  地址增长含奇数因子
  ↓
  组索引计算（位操作）分散分布
  ↓
  每 4096 行才回到同组
  ↓
  冲突极少（8 个 Way 够用）
```
