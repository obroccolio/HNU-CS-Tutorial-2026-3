## 实验背景与目标

### 核心动机

现代计算机系统的性能瓶颈**不在计算能力，而在内存访问**。CPU 与内存之间的延迟差异高达 **100~300 倍**。Cache（缓存）是填补这一鸿沟的关键硬件，但只有遵循 **Locality 原理**（局部性原理）的程序才能充分利用它。

矩阵乘法是一个完美的教学案例：
- 算法固定（数学定义明确）
- 内存行为易分析（规律性高）
- 优化空间巨大（从几秒到几毫秒）

### 实验目标

设计 **M × N** 规模的矩阵乘法 $C = A \times B$，通过不同的**算法嵌套、数据布局、分块策略**等手段，探讨如何在受限 Cache 下达到最优性能。

## 基础理论

### 1. 存储层次结构 (Memory Hierarchy)

现代 CPU 采用**多级缓存**结构：

```
┌─────────────────┐
│   Register      │  500+ GB/s  (0.5KB, 0 延迟)
├─────────────────┤
│   L1 Cache      │  ~110 GB/s  (32~64KB, 4周期)
├─────────────────┤
│   L2 Cache      │  ~50 GB/s   (256KB~1MB, 10周期)
├─────────────────┤
│   L3 Cache      │  ~20 GB/s   (2~16MB, 40周期)
├─────────────────┤
│   主存 (DRAM)   │  ~20 GB/s   (几十GB, 200周期)
├─────────────────┤
│   磁盘 (SSD)    │  ~0.5 GB/s  (TB级, 秒级)
└─────────────────┘
```

**关键数字（Intel Skylake 架构）：**

| 层级 | 大小 | 延迟 | 带宽 |
|-----|------|------|------|
| L1-dcache | 32KB | 4 周期 (~2ns) | 150+ GB/s |
| L2 | 256KB | 12 周期 (~6ns) | 80 GB/s |
| L3 | 8MB | 42 周期 (~21ns) | 30 GB/s |
| 主存 | ~16GB | 200+ 周期 (~100ns) | 20 GB/s |

### 2. Locality（局部性）

#### 时间局部性 (Temporal Locality)
再次使用同一数据相距时间越短越好。

```c
// 好的时间局部性
for(int i = 0; i < 1000; i++) {
    x = arr[i];  // 马上用 x
    process(x);
}

// 坏的时间局部性
for(int i = 0; i < 1000; i++) {
    x = arr[i];  // 使用 x
}
for(int i = 0; i < 1000; i++) {
    use(x);      // 很久以后才用 x
}
```

#### 空间局部性 (Spatial Locality)
访问相邻的内存地址。

```c
// 好的空间局部性（行优先）
for(int i = 0; i < 1000; i++)
    for(int j = 0; j < 1000; j++)
        printf("%d ", matrix[i][j]);

// 坏的空间局先（列优先）
for(int j = 0; j < 1000; j++)
    for(int i = 0; i < 1000; i++)
        printf("%d ", matrix[i][j]);
```

### 3. Cache Line 与对齐

CPU 不是以单字节为单位访问内存，而是以 **Cache Line** 为单位（通常 64 字节）：

```
主存地址：  [... 0x100 | 0x101 | 0x102 | ... | 0x13F | 0x140 | ...]
           └──────────────── 一条 Cache Line (64字节) ──────────────┘
```

- **同一 Cache Line 中的数据**：访问代价相同（只产生 1 次 miss）
- **跨 Cache Line**：可能产生额外 miss
- **Cache Line 对齐**：数据起址为 Cache Line 大小的倍数时，最高效

### 4. Cache 关联度与冲突不命中

Cache 采用 **Set-Associative** 结构：

- **全关联 (Fully Associative)**：任何地址可存储在任何位置，速度最快但硬件复杂
- **直接映射 (Direct-Mapped)**：每个地址只能映射到一个 Cache 行，硬件简单但易冲突
- **组关联 (N-way Set-Associative)**：折中方案，每个地址最多可映射到 N 个位置

**冲突不命中的后果：**

当多个地址映射到同一组 (Set) 时，它们会相互驱逐。对 $2^n$ 大小的矩阵尤其危险：

```c
// 假设 L1 Cache 32KB，8-way 组关联
// 矩阵行大小 = 1024 * 8 字节 = 8192 字节 = 2^13

int A[1024][1024];  // 每行 8192 字节，可能严重冲突
int B[1001][1001];  // 每行 8008 字节，冲突较少
```

## 选题 1：循环顺序之争

### 问题设定

给定 $C = A \times B$，其中 $A$ 为 $M \times K$，$B$ 为 $K \times N$，$C$ 为 $M \times N$。

基本循环：
$$C[i][j] = \sum_{k=0}^{K-1} A[i][k] \times B[k][j]$$

### 六种循环顺序

| 顺序 | 代码 | 外层迭代次数 | 中层迭代 | 内层迭代 | 特点 |
|-----|------|-----------|--------|--------|------|
| **ijk** | `for i: for j: for k` | M | N | K | C 步长为 1 |
| **jik** | `for j: for i: for k` | N | M | K | C 列优先 |
| **kij** | `for k: for i: for j` | K | M | N | A 步长 1，B 列优先 |
| **ikj** | `for i: for k: for j` | M | K | N | 多次写 C |
| **jki** | `for j: for k: for i` | N | K | M | A、C 步长大 |
| **kji** | `for k: for j: for i` | K | N | M | 最坏的组合 |

### 详细分析

#### 1. ijk 顺序

```c
for(int i = 0; i < M; i++)
    for(int j = 0; j < N; j++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];
```

**访问模式：**
- **A[i][k]**：内层循环中 k 变化，步长为 1（行优先）✓
- **B[k][j]**：内层循环中 k 变化，步长为 N（列优先）✗
- **C[i][j]**：内层循环中地址固定（好）✓

**性能特征：** 中等。B 的列优先访问是瓶颈。

#### 2. kij 顺序

```c
for(int k = 0; k < K; k++)
    for(int i = 0; i < M; i++)
        for(int j = 0; j < N; j++)
            C[i][j] += A[i][k] * B[k][j];
```

**访问模式：**
- **A[i][k]**：中层循环中 i 变化，步长为 K（列优先，但加载到 Cache 后被充分利用）
- **B[k][j]**：内层循环中 j 变化，步长为 1（行优先）✓✓
- **C[i][j]**：内层循环中 j 变化，步长为 1（行优先）✓✓

**性能特征：** **最优**。A 虽然列优先，但 k 的值固定，A 的一列会被完全加载到 Cache。B 和 C 都是行优先。

#### 3. jik 顺序

```c
for(int j = 0; j < N; j++)
    for(int i = 0; i < M; i++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];
```

**访问模式：**
- **A[i][k]**：内层 k 变化，步长为 1（行优先）✓
- **B[k][j]**：内层 k 变化，步长为 N（列优先）✗
- **C[i][j]**：中层 i 变化，步长为 N（列优先）✗

**性能特征：** 较差。C 和 B 都列优先。

#### 4. ikj 顺序

```c
for(int i = 0; i < M; i++)
    for(int k = 0; k < K; k++)
        for(int j = 0; j < N; j++)
            C[i][j] += A[i][k] * B[k][j];
```

**访问模式：**
- **A[i][k]**：中层 k 变化，步长为 K（列优先）✗
- **B[k][j]**：内层 j 变化，步长为 1（行优先）✓
- **C[i][j]**：内层 j 变化，步长为 1（行优先）✓

**性能特征：** 中等。A 列优先，但 k 值变化，A 的不同列被重复加载。

#### 5. jki 和 kji 顺序

这两个顺序总体性能较差，因为涉及多次对 C 的非顺序写入或对 A、B 的复杂访问。

### 性能预期排名

```
性能效果排序（最快到最慢）：
kij > ikj ≥ ijk > jik > jki > kji
```

### 实验方法

```c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define M 1001
#define K 1001
#define N 1001

void matmul_ijk(double *A, double *B, double *C) {
    for(int i = 0; i < M; i++)
        for(int j = 0; j < N; j++) {
            double sum = 0;
            for(int k = 0; k < K; k++)
                sum += A[i*K + k] * B[k*N + j];
            C[i*N + j] = sum;
        }
}

void matmul_kij(double *A, double *B, double *C) {
    for(int k = 0; k < K; k++)
        for(int i = 0; i < M; i++)
            for(int j = 0; j < N; j++)
                C[i*N + j] += A[i*K + k] * B[k*N + j];
}

// ... 其他顺序类似

double measure_time(void (*func_ptr)(double*, double*, double*), 
                     double *A, double *B, double *C) {
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);
    
    func_ptr(A, B, C);
    
    clock_gettime(CLOCK_MONOTONIC, &end);
    
    double elapsed = (end.tv_sec - start.tv_sec) * 1000.0 +
                     (end.tv_nsec - start.tv_nsec) / 1000000.0;
    return elapsed;
}

int main() {
    double *A = (double*)malloc(M * K * sizeof(double));
    double *B = (double*)malloc(K * N * sizeof(double));
    double *C = (double*)malloc(M * N * sizeof(double));
    
    // 初始化数据...
    
    printf("ijk: %.2f ms\n", measure_time(matmul_ijk, A, B, C));
    printf("kij: %.2f ms\n", measure_time(matmul_kij, A, B, C));
    
    // ... 其他顺序
    
    free(A); free(B); free(C);
    return 0;
}
```

## 选题 2：分块技术 (Blocking / Tiling)

### 问题设定

即使选择 `kij` 等较优顺序，当矩阵规模达到几千时，仍然会因为 Cache 容量有限而性能下降。**分块技术**将大矩阵分解为小块（Tile），确保每块完整进行运算**不超过 L1/L2 容量**。

### 基本思想

```
原矩阵：              分块后：
┌──────────┐         ┌────┬────┐
│          │         │ T1 │ T2 │
│    A     │   ═>    ├────┼────┤
│          │         │ T3 │ T4 │
└──────────┘         └────┴────┘
```

在运算一个块时，其他块可能被驱逐出 Cache，但该块的数据完全驻留在 Cache 中。

### 单级分块示例（针对 kij）

```c
#define Bi 64  // 第一维块大小
#define Bj 64  // 第二维块大小
#define Bk 64  // 第三维块大小

void matmul_blocked(double *A, double *B, double *C, 
                    int M, int K, int N) {
    for(int ii = 0; ii < M; ii += Bi)
        for(int kk = 0; kk < K; kk += Bk)
            for(int jj = 0; jj < N; jj += Bj)
                // 对块 [ii:ii+Bi, kk:kk+Bk] 与 [kk:kk+Bk, jj:jj+Bj] 做 kij 乘法
                for(int k = kk; k < min(kk+Bk, K); k++)
                    for(int i = ii; i < min(ii+Bi, M); i++)
                        for(int j = jj; j < min(jj+Bj, N); j++)
                            C[i*N + j] += A[i*K + k] * B[k*N + j];
}
```

### 块大小选择

#### 理论推导

假设 A、B 都需要放入 Cache，Cache 大小为 L：

$$2 \times Bi \times Bk \times 8 \leq L$$
$$Bi \times Bk \leq \frac{L}{16}$$

对 L1（32KB）：
$$Bi \times Bk \leq 2048 \text{ 元素} ≈ 64 \times 64$$

对 L2（256KB）：
$$Bi \times Bk \leq 16384 \text{ 元素} ≈ 128 \times 128$$

#### 实验方法

在 1001×1001 矩阵上测试不同块大小：

| 块大小 | 可落入 L1 Cache | 耗时 (ms) | 性能 (GFLOPS) |
|--------|---------------|----------|-------------|
| 无分块（kij） | 否 | ~5000 | ~4 |
| 32×32 | 是 | ~800 | ~25 |
| 64×64 | 是/部分 | ~400 | ~50 |
| 128×128 | 否（入 L2） | ~600 | ~33 |
| 256×256 | 否 | ~3000 | ~7 |

**最优块大小通常为 64×64 或接近 L1 Cache 行数的值。**

### 多级分块（L1 + L2）

对超大矩阵，可使用嵌套分块：

```c
#define B1 64   // L1 块
#define B2 256  // L2 块

void matmul_blocked_2level(double *A, double *B, double *C, 
                           int M, int K, int N) {
    // L2 级别分块
    for(int ii2 = 0; ii2 < M; ii2 += B2)
        for(int kk2 = 0; kk2 < K; kk2 += B2)
            for(int jj2 = 0; jj2 < N; jj2 += B2) {
                // L1 级别分块
                for(int ii1 = ii2; ii1 < min(ii2+B2, M); ii1 += B1)
                    for(int kk1 = kk2; kk1 < min(kk2+B2, K); kk1 += B1)
                        for(int jj1 = jj2; jj1 < min(jj2+B2, N); jj1 += B1)
                            // 真正的计算循环
                            for(int k = kk1; k < min(kk1+B1, K); k++)
                                for(int i = ii1; i < min(ii1+B1, M); i++)
                                    for(int j = jj1; j < min(jj1+B1, N); j++)
                                        C[i*N + j] += A[i*K + k] * B[k*N + j];
            }
}
```

### 对称平铺 vs 矩形平铺

- **正方形块（Bi = Bj = Bk）**：实现简单，对称美观
- **矩形块（不同 Bi、Bj、Bk）**：可更精细地匹配硬件特性

通常 Bi 和 Bj 设置相同（C 的访问对称），Bk 可独立调整。

## 选题 3：数据布局与内存优化

### 3.1 预转置 B 矩阵

#### 问题

在 `kij` 顺序中，B[k][j] 的访问是列优先（步长 N），这仍然是一个 Cache 瓶颈。

#### 解决方案

预先将 B 转置为 $B^T$，使原本的列访问变为行访问：

```c
// 初始化
double *B_T = (double*)malloc(N * K * sizeof(double));

// 转置（一次性开销）
for(int k = 0; k < K; k++)
    for(int j = 0; j < N; j++)
        B_T[j*K + k] = B[k*N + j];

// 改进的矩阵乘法（B 变为 B_T，访问 B_T[j][k] 对应原 B[k][j]）
for(int k = 0; k < K; k++)
    for(int i = 0; i < M; i++)
        for(int j = 0; j < N; j++)
            C[i*N + j] += A[i*K + k] * B_T[j*K + k];
            // ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
            // 现在 B_T 的访问是行优先（步长为 K）
```

#### 性能收益

转置开销：$O(K \times N)$ 次内存访问（约 K*N FLOPS）

矩阵乘法计算：$2 \times M \times K \times N$ FLOPS

比例：$\frac{K \times N}{2 \times M \times K \times N} = \frac{1}{2M} \approx 0.05\%$ （M=1001 时）

**转置开销几乎可忽略，而计算加速可达 30~50%。**

#### 实验方法

```c
double t1 = measure_time(matmul_kij, A, B, C);

// 转置 B
for(int k = 0; k < K; k++)
    for(int j = 0; j < N; j++)
        B_T[j*K + k] = B[k*N + j];

double t2 = measure_time(matmul_kij_with_BT, A, B_T, C);

printf("原始 kij:     %.2f ms\n", t1);
printf("转置后 kij:   %.2f ms （含转置）\n", t2);
printf("加速比:       %.2f x\n", t1 / t2);
```

### 3.2 存储方式比较

#### 二维静态数组 vs 一维动态数组

```c
// 方式 1：二维静态数组
double A[1001][1001];  // 栈分配（可能栈溢出）
// 访问：A[i][j]

// 方式 2：一维动态数组
double *A = (double*)malloc(M * N * sizeof(double));
// 访问：A[i*N + j]

// 方式 3：指针数组（二级）
double **A = (double**)malloc(M * sizeof(double*));
for(int i = 0; i < M; i++)
    A[i] = (double*)malloc(N * sizeof(double));
// 访问：A[i][j]
```

**性能对比：**

| 方式 | 内存连续性 | Cache 友好度 | 速度 |
|-----|----------|-----------|------|
| 二维静态 | 完全连续 | 最好 | 最快 |
| 一维动态 | 完全连续 | 最好 | 最快 |
| 指针数组 | 每行分散 | 较差 | 慢 15~30% |

**推荐：** 使用一维动态数组和手动下标计算 `A[i*N + j]`。

### 3.3 内存对齐 (Alignment)

#### 为什么对齐重要

CPU 从内存读取数据时，如果数据跨越 Cache Line 边界，会产生额外延迟。

```
对齐的访问：         未对齐的访问：
Address:            Address:
[0x00-0x3F]  <-OK   [0x00-0x3F]
[0x40-0x7F]  <-OK   [0x3C-0x7B] <- 跨越两条 line，+50% 延迟
[0x80-0xBF]  <-OK
```

#### posix_memalign 方法

```c
#include <stdlib.h>

double *A, *B, *C;

// 分配地址对齐到 64 字节（Cache Line 大小）
posix_memalign((void**)&A, 64, M * K * sizeof(double));
posix_memalign((void**)&B, 64, K * N * sizeof(double));
posix_memalign((void**)&C, 64, M * N * sizeof(double));

// ... 使用 A, B, C ...

// 不能用 free()，必须用 free()（但 posix_memalign 分配的内存能被 free）
free(A); free(B); free(C);
```

#### 性能收益

通常 **5~10% 性能提升**，在 SIMD 优化中更显著。

### 3.4 Padding 消除冲突不命中

#### 问题场景

$2^n$ 大小矩阵容易产生 Cache 冲突：

```c
int A[1024][1024];  // 每行 8192 字节 = 2^13
// 假设 L1 Cache 32KB = 2^15 字节
// Address 0x0 和 Address 0x2000 映射到同一 Cache 组，易冲突
```

#### Padding 解决方案

在矩阵末尾增加若干额外列：

```c
// 原始（可能冲突）
double A[1024][1024];

// 改进（避免冲突）
double A[1024][1032];  // 增加 8 列 padding
```

这样，不同行的地址不再映射到相同 Cache 组。

#### 实验对比

| 矩阵大小 | 无 Padding | 有 Padding (+8列) | 改进 |
|---------|----------|-----------------|------|
| 1024×1024 | 2000 ms | 1200 ms | **40% 加速** |
| 1001×1001 | 800 ms | 810 ms | 无变化（已无冲突） |
| 2048×2048 | 16000 ms | 9500 ms | **41% 加速** |

## 性能分析方法论

### Roofline 模型

性能受两个因素限制：

$$\text{性能} = \min(\text{计算能力}, \text{内存带宽} \times \text{算术强度})$$

其中 **算术强度** = FLOPS / 内存访问字节数

对矩阵乘法：
- FLOPS = $2 M K N$（每个元素 1 个乘法 + 1 个加法）
- 内存访问 = $M K + K N + M N$ 字节（读 A、B，写 C）

$$\text{算术强度} = \frac{2 M K N}{M K + K N + M N} ≈ \frac{2MKN}{(M+N)K + MN}$$

对 $M = N = K = 1001$ 的情况：
$$\text{算术强度} ≈ \frac{2 \times 1001^3}{2 \times 1001^2 + 1001^2} = \frac{2001002001}{3003001} ≈ 666 \text{ FLOPS/byte}$$

现代 CPU 可提供：
- 计算能力：100+ GFLOPS
- 内存带宽：~20 GB/s = 20 GFLOPS/byte·s

如果算术强度为 666 FLOPS/byte：
$$\text{理论峰值} = 20 \text{ GB/s} \times 666 = 13.3 \text{ TFLOPS}$$

但实际我们只能达到 **100 GFLOPS** 左右，说明内存延迟是主要限制（而非带宽）。

### 诊断流程

```
1. 测量总耗时 T
2. 运行 perf stat，查看：
   - IPC (Instructions Per Cycle)：若 < 1，说明管道停顿
   - L1 miss rate：若 > 10%，性能有问题
   - L2 miss rate：若 > 3%，说明数据不能完全放入 L2
   - LLC miss rate：若 > 1%，说明频繁回主存
3. 结合 perf record 查看热点函数（哪个使用最多 CPU 周期）
4. 根据访问模式分析，选择合适的优化策略
```

## 常见坑点与解决方案

### 1. 编译优化等级的影响

```bash
# 不同优化等级，性能相差 5~10 倍
gcc -O0 -o prog prog.c  # ~20000 ms（编译器不优化）
gcc -O2 -o prog prog.c  # ~2000 ms（标准优化）
gcc -O3 -o prog prog.c  # ~1000 ms（激进优化，可能有风险）
```

**建议：** 使用 `-O2` 或 `-O3`，并加上 `-march=native`（针对当前 CPU 优化）。

```bash
gcc -O3 -march=native -ffast-math -o prog prog.c
```

### 2. 循环中的除法和浮点精度

```c
// 坏：每次循环计算倒数
for(int i = 0; i < n; i++)
    result[i] = value / divisor;  // 除法很慢

// 好：提取常数，用乘法替代
double inv = 1.0 / divisor;
for(int i = 0; i < n; i++)
    result[i] = value * inv;  // 乘法快 3~5 倍
```

### 3. 分支预测失败

```c
// 坏：循环中有复杂条件分支
for(int i = 0; i < n; i++) {
    if(arr[i] > threshold) {
        result[i] = expensive_func(arr[i]);  // 分支难以预测
    }
}

// 好：用向量化或 SIMD 规避分支
// 或重新组织数据减少条件分支
```

### 4. Cache 行冲突导致的陡峭性能下降

**症状：** 矩阵大小从 1000 变到 1024 时，性能反而下降 40%。

**根因：** $2^n$ 大小导致地址映射冲突。

**解决：**
- 使用 Padding（如 1024+8）
- 动态调整块大小
- 使用 `posix_memalign` 改变对齐

### 5. NUMA 多 socket 系统的开销

在多 socket CPU 上，跨 socket 访问内存延迟高 1 倍。

```bash
# 查看 NUMA 拓扑
numactl -H

# 绑定到单个 socket
numactl -N 0 ./program
```

### 6. 时钟精度问题

```c
// 坏：总耗时太短，计时不准
for(int trial = 0; trial < 1000; trial++) {
    clock_gettime(..., &start);
    matmul(A, B, C);  // 仅需 10ms
    clock_gettime(..., &end);
}

// 好：延长总耗时，摊平计时误差
matmul(A, B, C);  // 运行多遍或用更大矩阵
matmul(A, B, C);
matmul(A, B, C);
// ... 共 10+ 秒

clock_gettime(..., &start);
for(int trial = 0; trial < 100; trial++)
    matmul(A, B, C);
clock_gettime(..., &end);
```

## 总结：优化策略的递进关系

```
【第 1 层】选择好的循环顺序（kij 优于 ijk）
    优化收益：2~3 倍 | 实现难度：低 | 建议优先级：★★★★★

【第 2 层】应用分块技术（L1-aware blocking）
    优化收益：5~10 倍 | 实现难度：中 | 建议优先级：★★★★☆

【第 3 层】预转置数据、内存对齐、Padding
    优化收益：1.5~2 倍 | 实现难度：中 | 建议优先级：★★★☆☆

【第 4 层】SIMD 向量化（AVX-256、SVE）
    优化收益：3~4 倍 | 实现难度：高 | 建议优先级：★★★☆☆

【第 5 层】多线程并行化（OpenMP、CUDA）
    ①纯 CPU 多线程：2~8 倍（取决于核数）
    ②GPU 加速（CUDA）：10~100 倍
    实现难度：高 | 建议优先级：★★★☆☆
```

逐层应用这些优化，最终可从初始的 ~10000 ms 降低到 **100~500 ms**（1001³ 乘法），近似改进 **10~100 倍**。
