# 虚拟机环境下顺序访问与随机访问对比、宿主机差异实验操作手册

## 一、实验目标

本手册围绕 3.1.c 的工作集大小扫描实验进行扩展，重点研究两个问题：

1. 在虚拟机中，顺序访问和随机访问的平均访问延迟有什么差异？
2. 虚拟机中测得的缓存边界、访问延迟是否与宿主机一致？如果不一致，原因是什么？

这两个扩展仍然服务于 3.1.c 的核心问题：通过工作集大小扫描观察 L1、L2、L3、内存之间的容量边界，并解释实验结果与 `lscpu` 报告之间的差异。

## 三、扩展 1：虚拟机中顺序访问与随机访问对比

### 1. 实验目的

比较两种访问模式：

```text
顺序访问：按数组地址从小到大访问
随机访问：随机 pointer chasing，每次访问依赖上一次访问结果
```

顺序访问容易被 CPU 硬件预取器优化，因此平均延迟通常更低、曲线更平滑；随机访问难以预取，更容易暴露 L1、L2、L3、内存的容量边界。

### 2. 准备对比程序

新建文件 `wss_compare.c`：

```c
#define _POSIX_C_SOURCE 200809L

#include <stdint.h>
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define MIN_SIZE   (4ULL * 1024ULL)
#define MAX_SIZE   (1ULL * 1024ULL * 1024ULL * 1024ULL)
#define CACHE_LINE 64ULL
#define REPEATS    5
#define NS_PER_S   1000000000.0

static inline double elapsed_ns(struct timespec a, struct timespec b)
{
    return (b.tv_sec - a.tv_sec) * NS_PER_S + (b.tv_nsec - a.tv_nsec);
}

static inline void *slot_addr(unsigned char *base, size_t i)
{
    return base + i * CACHE_LINE;
}

__attribute__((noinline))
static double measure_sequential(unsigned char *buf, size_t size, long long total)
{
    struct timespec t0, t1;
    volatile unsigned long long sum = 0;
    size_t slots = size / CACHE_LINE;

    clock_gettime(CLOCK_MONOTONIC, &t0);
    for (long long k = 0; k < total; k++) {
        size_t idx = (size_t)k % slots;
        sum += buf[idx * CACHE_LINE];
    }
    clock_gettime(CLOCK_MONOTONIC, &t1);

    asm volatile("" : : "r"(sum) : "memory");
    return elapsed_ns(t0, t1) / (double)total;
}

__attribute__((noinline))
static double measure_random(void *start, long long total)
{
    struct timespec t0, t1;
    void *p = start;

    for (int i = 0; i < 1000; i++) {
        asm volatile("" : "+r"(p) : : "memory");
        p = *(void **)p;
    }

    clock_gettime(CLOCK_MONOTONIC, &t0);
    for (long long k = 0; k < total; k++) {
        asm volatile("" : "+r"(p) : : "memory");
        p = *(void **)p;
    }
    clock_gettime(CLOCK_MONOTONIC, &t1);

    asm volatile("" : : "r"(p) : "memory");
    return elapsed_ns(t0, t1) / (double)total;
}

static int cmp_double(const void *a, const void *b)
{
    double x = *(const double *)a;
    double y = *(const double *)b;
    return (x > y) - (x < y);
}

static void print_size(size_t size)
{
    const char *unit = "B";
    double value = (double)size;

    if (size >= 1024ULL * 1024ULL * 1024ULL) {
        value /= 1024.0 * 1024.0 * 1024.0;
        unit = "GB";
    } else if (size >= 1024ULL * 1024ULL) {
        value /= 1024.0 * 1024.0;
        unit = "MB";
    } else if (size >= 1024ULL) {
        value /= 1024.0;
        unit = "KB";
    }

    printf("%8.0f %-2s", value, unit);
}

int main(void)
{
    srand((unsigned)time(NULL));

    printf("%-12s %16s %16s\n", "WorkingSet", "Sequential(ns)", "Random(ns)");
    printf("%-12s %16s %16s\n", "----------", "--------------", "----------");

    for (size_t size = MIN_SIZE; size <= MAX_SIZE; size *= 2) {
        size_t slots = size / CACHE_LINE;
        unsigned char *buf = malloc(size);
        size_t *perm = malloc(slots * sizeof(*perm));
        long long total = (long long)slots * 5LL;
        double seq[REPEATS];
        double rnd[REPEATS];

        if (!buf || !perm) {
            fprintf(stderr, "malloc failed at size %zu\n", size);
            free(buf);
            free(perm);
            break;
        }

        if (total < 2000000LL) total = 2000000LL;
        if (total > 10000000LL) total = 10000000LL;

        for (size_t i = 0; i < size; i++) {
            buf[i] = (unsigned char)i;
        }

        for (size_t i = 0; i < slots; i++) {
            perm[i] = i;
        }

        for (size_t i = slots - 1; i > 0; i--) {
            size_t j = (size_t)rand() % (i + 1);
            size_t tmp = perm[i];
            perm[i] = perm[j];
            perm[j] = tmp;
        }

        for (size_t i = 0; i < slots; i++) {
            size_t cur = perm[i];
            size_t next = perm[(i + 1) % slots];
            *(void **)slot_addr(buf, cur) = slot_addr(buf, next);
        }

        for (int r = 0; r < REPEATS; r++) {
            seq[r] = measure_sequential(buf, size, total);
            rnd[r] = measure_random(slot_addr(buf, 0), total);
        }

        qsort(seq, REPEATS, sizeof(seq[0]), cmp_double);
        qsort(rnd, REPEATS, sizeof(rnd[0]), cmp_double);

        print_size(size);
        printf(" %16.2f %16.2f\n", seq[REPEATS / 2], rnd[REPEATS / 2]);

        free(perm);
        free(buf);
    }

    return 0;
}
```

### 5. 结果分析要点

观察图像时重点回答：

1. 顺序访问曲线是否明显低于随机访问曲线？
2. 顺序访问是否更平滑，随机访问是否更容易出现台阶？
3. 随机访问的转折点是否接近 L1d、L2、L3 容量？
4. 虚拟机中随机访问延迟是否偏高？

可写入报告的分析：

```text
顺序访问由于地址连续，CPU 硬件预取器可以提前加载后续 cache line，因此平均访问延迟较低。
随机访问采用 pointer chasing，每一次访问地址依赖上一次读取结果，CPU 难以提前预取，因此更能反映缓存层级切换带来的延迟变化。
在虚拟机中，随机访问延迟可能进一步升高，因为虚拟 CPU 调度、宿主机后台负载和虚拟内存管理都会引入额外抖动。
```

### 5. 差异分析

虚拟机与宿主机结果不一致是正常现象，常见原因如下：

1. 虚拟 CPU 调度  
   虚拟机中的 vCPU 由宿主机调度，实验进程可能被暂停或迁移，导致延迟抖动。

2. Cache 不独占  
   即使虚拟机看到 L3 Cache 容量，实际 L3 通常与宿主机和其他虚拟机共享。

3. 内存虚拟化开销  
   虚拟地址到物理地址之间多了一层映射，TLB miss 和页表访问可能更复杂。

4. 宿主机后台负载  
   宿主机上的浏览器、IDE、杀毒软件、其他虚拟机等都会影响 Cache 和内存带宽。

5. CPU 频率变化  
   虚拟机内看到的时间和 CPU 频率可能受宿主机电源策略、睿频、温度限制影响。

6. NUMA 和内存分配  
   如果宿主机是多 NUMA 节点机器，虚拟机内存分布可能影响访问延迟。

### 2. 虚拟机已记录 Cache 信息

本实验主要采用 /sys/devices/system/cpu/cpu0/cache/index*/size 的结果作为缓存边界：
L1d = 48KB，L2 = 2MB，L3 = 36MB。

原因是本实验程序主要是单线程随机访问，实际更接近单个逻辑 CPU 视角下的缓存层级。
lscpu 输出的 L1d=96KB、L2=4MB、L3=72MB 可能是虚拟机 2 个 vCPU 的汇总值，因此只作为环境信息记录，不作为曲线转折点判断的主要依据。

根据 Ubuntu 虚拟机中 `/sys/devices/system/cpu/cpu0/cache/index*/size` 的输出：

```text
index0: 48K      L1d 数据缓存
index1: 32K      L1i 指令缓存
index2: 2048K    L2 缓存，即 2 MB
index3: 36864K   L3 缓存，即 36 MB
```

因此，本实验分析数据访问时主要采用：

```text
L1d = 48 KB
L2  = 2 MB
L3  = 36 MB
```

从虚拟机工作集扫描图中可以观察到：工作集从 4KB 增长到 1GB 时，随机访问平均延迟整体上升；在约 64KB 附近开始离开 L1d 的低延迟区，在 2MB 之后继续升高，超过几十 MB 后进入更高的内存访问延迟区。这与 `L1d = 48KB`、`L2 = 2MB`、`L3 = 36MB` 的缓存层级边界基本吻合。

### 4. 顺序访问与随机访问结果分析

宿主机结果显示，顺序访问延迟整体非常稳定，大部分工作集大小下保持在约 2 ns 到 3 ns 之间。即使工作集增长到 1GB，顺序访问平均延迟也只有约 3.03 ns。这说明顺序访问被硬件预取器很好地优化，CPU 可以提前把后续 cache line 加载进缓存，因此曲线不容易表现出明显的 L1/L2/L3 分界。

随机访问结果则明显不同。4KB 到 32KB 时随机访问延迟约 0.97 ns，仍处于很低的缓存命中区；到 64KB 时延迟上升到约 3.12 ns，说明工作集已经超过 48KB 的 L1d 数据缓存边界；到 2MB 时延迟约 6.50 ns，开始接近 L2 边界；4MB 后延迟上升到 17.25 ns，说明更多访问需要进入更远的缓存层级；16MB、32MB、64MB 之后延迟继续升高，最终 1GB 达到约 110.94 ns，表现出进入主内存访问区域后的高延迟。

因此，顺序访问适合观察硬件预取带来的性能优势，而随机 pointer chasing 更适合本题 3.1.c 的目标，即观察工作集大小超过 L1、L2、L3 后平均访问延迟的台阶变化。

### 5. 虚拟机与宿主机差异分析

本机虚拟机和宿主机的 L3 Cache 信息一致，均可观察到约 36MB 的 L3 层级；虚拟机中记录的 L1d 为 48KB，L2 为 2MB，符合 i9-13900HX 单核附近的数据缓存层级特征。

但虚拟机实验图中的随机访问绝对延迟明显高于宿主机实测。例如，宿主机随机访问 1GB 工作集约为 110.94 ns，而虚拟机图中 1GB 工作集已经达到数百 ns。该现象不代表实验错误，而是虚拟机环境中常见的额外开销。

主要原因包括：

1. 虚拟机只有 2 个 vCPU，实际运行时由 Windows 宿主机调度，实验线程可能被暂停或迁移。
2. L3 Cache 是宿主机多个核心共享的资源，虚拟机不能独占 L3。
3. 虚拟机内存为 4GB，运行 1GB 工作集时更容易受到内存压力、页表和 TLB miss 的影响。
4. Windows 宿主机后台程序、VMware 本身、图形界面和杀毒/安全服务都可能抢占 CPU 和内存带宽。
5. 虚拟地址到宿主机物理地址之间存在额外映射层，随机访问更容易放大这类开销。

所以，虚拟机实验适合用于观察“转折点”和“趋势”，即能否看见 L1/L2/L3/内存边界；但不适合把虚拟机中的绝对 ns 数值当作裸机性能。

### 6. 可直接写入报告的结论

```text
本扩展实验比较了顺序访问和随机访问在工作集大小扫描中的表现。宿主机实测结果显示，顺序访问延迟基本保持在 2ns 到 3ns 左右，随着工作集增大变化较小；随机访问则在 64KB、2MB、几十 MB 等位置出现明显上升，更容易反映 L1d、L2、L3 和内存之间的层级差异。

Ubuntu 虚拟机中记录到的缓存信息为 L1d 48KB、L2 2MB、L3 36MB，工作集扫描图中的转折趋势与这些容量边界基本吻合。但虚拟机中的绝对延迟高于宿主机，尤其在大工作集随机访问时更明显。这主要与虚拟 CPU 调度、L3 共享、虚拟内存映射、宿主机后台负载以及虚拟机内存限制有关。

因此，在虚拟机中进行 3.1.c 实验时，可以使用随机访问曲线观察缓存容量边界；但分析时应重点关注曲线趋势和转折点，而不是将虚拟机中的绝对延迟直接等同于宿主机裸机性能。
```
