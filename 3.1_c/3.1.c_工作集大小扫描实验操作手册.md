# 3.1.c 工作集大小扫描实验操作手册

## 一、实验目的

本实验对应题目 3.1.c：使用“工作集大小扫描”方法，将数组大小从 4KB 到 1GB 按 2 倍指数增长，测量每次内存访问的平均耗时，观察访问延迟随工作集大小变化的转折点。

通过实验回答：

1. 能否从访问延迟曲线中清晰看见 L1、L2、L3、内存的容量边界？
2. 曲线转折点是否与 `lscpu` 报告的 Cache 容量吻合？
3. 如果不完全吻合，可能由哪些因素导致？

适当扩展：在完成基本扫描的基础上，增加多次重复、取中位数、绘图、与顺序访问对比、使用 `perf` 观察 cache miss 等操作，帮助解释转折点是否可信。

## 二、实验原理

CPU 访问数据时会优先从最近的缓存层级读取。一般访问路径为：

```text
寄存器 -> L1 Cache -> L2 Cache -> L3 Cache -> 内存
```

越靠近 CPU，容量越小、延迟越低；越靠近内存，容量越大、延迟越高。

本实验逐步增大工作集大小：

```text
4KB, 8KB, 16KB, 32KB, ..., 512MB, 1GB
```

当工作集能放进 L1 Cache 时，大多数访问命中 L1，平均访问延迟较低；当工作集超过 L1 容量后，访问会更多落到 L2，延迟上升。类似地，超过 L2、L3 后，会继续出现延迟上升。

为了避免 CPU 硬件预取器把顺序访问优化掉，实验程序采用随机指针追踪：

```c
p = *(void **)p;
```

每次访问的下一个地址由当前内存内容决定，CPU 难以提前预取，因此更容易测出真实的缓存层级边界。

```bash
lscpu | grep -i cache
```


如果是虚拟机，`lscpu` 看到的 Cache 信息可能来自宿主机透传，也可能被虚拟化环境处理过，需要在分析中说明。

### 2. 编译并运行程序

```bash
gcc -O2 -Wall -Wextra -std=gnu11 wss_scan.c -o wss_scan
./wss_scan | tee wss_result.txt
```

建议关闭明显占用 CPU 或内存的程序，避免结果抖动。

### 3. 重复运行多次

为了减少偶然误差，建议至少运行 3 次：

```bash
./wss_scan | tee wss_result_1.txt
./wss_scan | tee wss_result_2.txt
./wss_scan | tee wss_result_3.txt
```

分析时优先观察每个工作集大小下的中位数或趋势，而不是只看单次结果。

### 4. 绘制曲线

推荐横轴使用对数刻度，因为工作集大小按 2 倍增长。

可以使用 Python 绘图。新建 `plot_wss.py`：

```python
import re
import matplotlib.pyplot as plt

unit_scale = {
    "B": 1,
    "KB": 1024,
    "MB": 1024 ** 2,
    "GB": 1024 ** 3,
}

sizes = []
latencies = []

with open("wss_result.txt", "r", encoding="utf-8") as f:
    for line in f:
        m = re.match(r"\s*([0-9.]+)\s+(B|KB|MB|GB)\s+([0-9.]+)\s+ns", line)
        if not m:
            continue
        size = float(m.group(1)) * unit_scale[m.group(2)]
        latency = float(m.group(3))
        sizes.append(size)
        latencies.append(latency)

plt.figure(figsize=(8, 5))
plt.plot(sizes, latencies, marker="o")
plt.xscale("log", base=2)
plt.xlabel("Working Set Size (bytes, log2)")
plt.ylabel("Average Access Latency (ns)")
plt.title("Working Set Size Scan")
plt.grid(True, which="both", linestyle="--", alpha=0.5)
plt.tight_layout()
plt.savefig("wss_scan.png", dpi=200)
plt.show()
```

运行：

```bash
python3 plot_wss.py
```

最终得到 `wss_scan.png`。



## 六、结果分析方法

### 1. 标出理论边界

在图中或表格中标出 `lscpu` 报告的容量：

```text
L1d: ______
L2 : ______
L3 : ______
```

然后观察延迟明显上升的位置。例如：

```text
4KB ~ 32KB: 延迟较低，可能主要命中 L1d
64KB ~ 1MB: 延迟升高，可能进入 L2
2MB ~ 32MB: 延迟继续上升，可能进入 L3
64MB 以上: 延迟明显增大，更多访问内存
```

注意：转折点通常不是一条垂直线，而是一段过渡区域。

### 2. 判断是否吻合

可以使用如下表格：

| 层级 | `lscpu` 容量 | 曲线观察到的转折点 | 是否基本吻合 | 说明 |
| --- | --- | --- | --- | --- |
| L1d | ____ | ____ | 是/否 | ____ |
| L2 | ____ | ____ | 是/否 | ____ |
| L3 | ____ | ____ | 是/否 | ____ |
| 内存 | 超过 L3 后 | ____ | 是/否 | ____ |

若曲线转折点略大于或略小于 `lscpu` 容量，通常仍可认为基本吻合。

### 3. 解释偏差原因

可能原因包括：

1. Cache 组相联限制：即使总容量够，随机访问也可能因为映射冲突导致提前 miss。
2. TLB 影响：工作集变大后，页表项和 TLB miss 会额外增加延迟。
3. 虚拟机影响：虚拟机中 Cache 信息和实际调度可能受宿主机影响。
4. 后台负载：其他进程会抢占 CPU、Cache 和内存带宽。
5. 频率变化：CPU 睿频、节能策略会改变测量时间。
6. L3 共享：L3 往往被多个核心共享，其他核心活动会影响结果。

## 七、切合题目的扩展实验

### 扩展 1：顺序访问与随机访问对比

目的：说明为什么本题适合使用随机指针追踪。

做法：另外写一个顺序遍历数组的程序，按 cache line 顺序访问：

```c
for (size_t i = 0; i < size; i += 64) {
    sum += buf[i];
}
```

预期现象：

```text
顺序访问曲线更平滑，延迟可能较低；
随机 pointer chasing 曲线更容易暴露 L1/L2/L3/内存边界。
```

原因：顺序访问会触发硬件预取器，CPU 可以提前把后续数据拉入 Cache。

### 扩展 2：使用 perf 验证 cache miss

目的：让“看见边界”不仅依赖耗时曲线，也有硬件计数器支持。

运行：

```bash
perf stat -e cycles,instructions,cache-references,cache-misses ./wss_scan
```

关注：

```text
cache-misses
cache-miss rate
cycles
```

当工作集超过 L1/L2/L3 后，cache miss 比例通常会上升。

说明：不同机器支持的 perf 事件名称可能不同。如果上述命令失败，可先运行：

```bash
perf list | grep -i cache
```

### 扩展 3：固定 CPU 核心，减少调度干扰

目的：降低程序在不同核心之间迁移带来的 Cache 变化。

运行：

```bash
taskset -c 0 ./wss_scan | tee wss_result_taskset.txt
```

对比普通运行和固定核心运行的结果。如果固定核心后曲线更稳定，可在报告中说明。

### 扩展 4：观察虚拟机与宿主机差异

如果实验在虚拟机中完成，可以记录：

```bash
systemd-detect-virt
lscpu
```

分析虚拟机环境可能带来的影响：

```text
虚拟机中的 CPU 核心可能被宿主机调度；
Cache 信息可能来自宿主机，但实验进程不一定独占 Cache；
内存访问可能受虚拟化层、NUMA、宿主机负载影响。
```

该扩展仍然紧扣 3.1.c，因为它解释“转折点是否与 lscpu 吻合”这一问题。

## 八、实验报告建议结构

实验报告可按以下结构撰写：

1. 实验目的：说明要通过工作集扫描观察存储层次边界。
2. 实验环境：列出 CPU、Cache、内存、操作系统、编译器版本。
3. 实验方法：说明数组大小范围、指数增长、随机指针追踪、重复测量取中位数。
4. 实验结果：给出原始表格和延迟曲线图。
5. 边界分析：将曲线转折点与 `lscpu` 的 L1d/L2/L3 容量对照。
6. 扩展验证：加入顺序访问对比、`perf` 或固定 CPU 核心结果。
7. 结论：回答是否能清晰看见 L1/L2/L3/内存边界，以及是否与 `lscpu` 吻合。

## 九、结论模板

可参考如下写法：

```text
本实验使用工作集大小扫描方法，将随机访问数组大小从 4KB 增长到 1GB。
实验结果显示，平均访问延迟并非线性增长，而是在若干工作集大小附近出现明显台阶。
这些台阶大致对应 L1d、L2、L3 Cache 容量边界。

与 lscpu 报告的 Cache 容量相比，L1d 和 L2 的转折点较容易观察；
L3 到内存的边界也较明显，但转折区域更宽。
偏差主要可能来自 Cache 组相联冲突、TLB miss、虚拟机调度、后台负载和 CPU 频率变化。

顺序访问对比实验表明，由于硬件预取器作用，顺序访问曲线更平滑；
随机指针追踪更适合用于观察缓存容量边界。
```
