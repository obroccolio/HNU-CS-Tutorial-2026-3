# 题目 d) 转置 B 矩阵的性能影响

## 🎯 通俗解释

### 问题的核心

你知道"行优先"和"列优先"吗？简单说：
- **行优先**：相邻的数据在内存中挨在一起，CPU 能一口气读出来 ✓ 快
- **列优先**：数据在内存中散开，CPU 要跑来跑去读，满地跑 ✗ 慢

C 语言数组是**行优先**的：
```c
int A[1000][1000];  
// A[0][0] A[0][1] A[0][2] ... A[0][999] A[1][0] A[1][1] ...
// ^ 内存中连续排列
```

### 矩阵乘法中 B 的访问模式

```c
for(int i = 0; i < M; i++)
    for(int j = 0; j < N; j++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];  // ← 看这里
```

当 k 变化时，B[k][j] 的访问是这样的：
```
B[0][j], B[1][j], B[2][j], ...
 ^       ^       ^
这些在内存中隔很远（间隔为 N 个元素）
```

**这就是列优先访问，很慢！**

### 转置的妙处

如果我们先把 B 转置（行变列，列变行）：

```c
原 B:           转置后 B_T:
B[k][j]   -->   B_T[j][k]

B[0][0] B[0][1]         B_T[0][0] B_T[1][0]
B[1][0] B[1][1]         B_T[0][1] B_T[1][1]
```

现在访问 `B_T[j][k]` 时，k 变化的部分变成**行优先**了，内存中就连续了！

### 转置值不值得？

转置的代价：$1001 \times 1001 \approx 100$ 万次内存操作（一次性）

矩阵乘法：$1001 \times 1001 \times 1001 \approx 10$ 亿次操作（频繁访问 B）

**转置开销** / **乘法开销** $\approx 0.1\%$，几乎忽略不计！

而 B 的访问模式改善后，**整体可能快 30~50%**，完全值得。

1. **性能提升约 30~50%**（取决于 CPU 型号和 Cache 大小）
2. **转置开销微不足道**（< 1% 的总时间）
3. **原因**：B 从列优先变为行优先，Cache 命中率大幅提升

### 访问模式详细对比

| 方面 | 版本 1（不转置） | 版本 2（转置后） |

| L1 Cache Miss | 频繁 | 较少 |

| 额外开销 | 0 | 转置 ~1% |

# 题目 e) 矩阵规模对性能的影响（Cache 冲突陷阱）

## 🎯 通俗解释

### 什么是 Cache 冲突？

想象 CPU 的 L1 Cache 是一个**有限的、分组的停车场**：

```
停车场有 8 排，每排 4 个位置（组关联）
某些汽车牌号计算结果总是对应到第 3 排

如果两辆车的牌号都对应第 3 排，后来的车就得把前面的车"赶出去"
即使其他排还有空位
```

这就是 **Cache 冲突** (Conflict Miss) 的本质。

### 为什么 2^n 大小容易冲突？

```c
int A[1024][1024];  // 2^10 × 2^10

// 每行占用 1024 × 8 = 8192 字节 = 2^13 字节

// L1 Cache 通常 32KB = 2^15 字节，8-way 组关联

// 地址 0x0 和 0x2000 (2^13) 会映射到同一 Cache 组
// A[0][0] 和 A[256][0] 不断互相驱逐

for(int i = 0; i < 1024; i++)
    for(int j = 0; j < 1024; j++)
        process(A[i][j]);  // ← 频繁 miss
```

### 为什么 1001 反而更快？

```c
int A[1001][1001];  // 不是 2^n

// 每行占用 1001 × 8 = 8008 字节（不规则）

// Address 映射比较"随机"，不容易同时冲突
// 多个数据能并存在 Cache 中

for(int i = 0; i < 1001; i++)
    for(int j = 0; j < 1001; j++)
        process(A[i][j]);  // ← miss 少得多
```

**讽刺的是：整齐的"2^n"反而更慢！**

### 性能曲线特征

```
性能（GFLOPS）
    ▲
    │     ▁▂▃▄▅    ▁▂▃▄▅▆▇    ▁▂▃▄▅
    │    ▁    ▔▔▔▔▔▔    ▔▔▔▔▔▔▔▔    
    │   ▁              
    │  ▁                
0.5 ├─▁─────────────┬──────────────┬──────────
    │               └─ 陡峭下跌     │
    │                 (1024 冲突)   │
0.3 │                              └─ 陡峭下跌
    │                                (2048 冲突)
    └────┬────┬────┬────┬────┬────┬────┬────―→
       1000  1024  1050      2000  2048  规模
```

### 关键观察

✅ **性能曲线不是平滑的，而是有陡峭的跳变：**

1. **1001 → 1024**：性能在 1024 处**陡峭下跌**（下降 30~50%）
2. **1025 → 2048**：性能逐渐恢复，直到 2048 再次出现跳变
3. **2048 之后**：性能可能继续下降或恢复

### 为什么 1024 这么特别？

| 规模 | 每行字节数 | 特征 | Cache 冲突 | 性能 |
|-----|----------|------|----------|------|
| 1001 | 8008 | 非 2^n（不规则） | 少 | 快 ✓ |
| 1024 | 8192 = 2^13 | **2^10 × 2^10** | 严重 | 慢 ✗ |
| 1025 | 8200 | 非 2^n（不规则） | 少 | 快 ✓ |
| 2048 | 16384 = 2^14 | **2^11 × 2^11** | 更严重 | 很慢 ✗✗ |

### 冲突的根本原因

Cache 采用"组关联"结构，其中：
- L1 Cache：32KB，8-way 关联
- Address 映射由地址的中间几位决定

当矩阵大小为 2^n 时：
- 多行数据的地址映射规律
- 它们同时竞争同一个 Cache 组
- 导致频繁的冲突不命中（Conflict Miss）

而非 2^n 规模则：
- Address 映射更"随机"
- 冲突较少
- 多行数据能共存于 Cache

### 解决方案（可选）

**方案 1：使用 Padding**
```c
// 原始（易冲突）
double A[1024][1024];

// 改进（避免冲突）
double A[1024][1032];  // +8 列填充
```

**效果**：破坏了 2^n 对齐，冲突消失，性能可恢复 30~50%

**方案 2：避免 2^n 规模**
```c
// 改用非 2^n 规模
double A[1001][1001];  // 原题要求
double A[2000][2000];  // 而不是 2048
```

## 额外注意事项

1. **运行时间差异**：由于 CPU、内存、系统负载等因素，你的结果可能与预期输出有所不同，但 **趋势应该相同**。

2. **编译优化**：必须使用 `-O3` 优化，否则看不到 Cache 效应。

3. **GFLOPS 计算**：
   $$\text{GFLOPS} = \frac{2 \times M \times K \times N}{耗时(秒)} \times 10^{-9}$$

4. **多核影响**：如果系统负载高，结果波动可能较大。建议在安静的系统上运行，或多次测试求平均。
