# 2.1 循环顺序之争 (Loop Permutations)

## 题目概述

通过实现不同嵌套循环顺序的矩阵乘法，探讨计算机缓存（Cache）对程序性能的影响。C 语言二维数组按**行优先（Row-major）** 存储，改变 `i, j, k` 三层循环的嵌套顺序会极大改变内存访问模式，从而影响 Cache 命中率和总体性能。

---

## 实验基础

### 矩阵乘法定义

设矩阵 $A$ 为 $M \times K$，矩阵 $B$ 为 $K \times N$，矩阵 $C$ 为 $M \times N$。

基本计算公式：
$$C[i][j] = \sum_{k=0}^{K-1} A[i][k] \times B[k][j]$$

### 矩阵规模

- 最小规模：**1001 × 1001**
- 可扩展规模：1024、1025、2048 等，用于观察 Cache 效应

---

## 推荐待回答问题

### a) 性能排序：六种循环顺序对比

**要求：**
- 实现六种循环嵌套顺序：`ijk`、`jik`、`kij`、`ikj`、`jki`、`kji`
- 在 1001×1001 矩阵规模下，分别测量每种顺序的运行时间
- 用表格和柱状图展示性能排序

**实现示例：**

```c
// ijk 顺序
for(int i = 0; i < M; i++)
    for(int j = 0; j < N; j++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];

// jik 顺序
for(int j = 0; j < N; j++)
    for(int i = 0; i < M; i++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];

// ... 其他四种顺序
```

**预期结果：** 行优先访问的顺序性能明显优于列优先，因为 Cache 行（Cache Line）通常为 64 字节，行优先步长为 1 最高效。

---

### b) 访问模式分析

**要求：**
- 对每一种循环顺序，分析矩阵 A、B、C 在内存中的访问模式
- 分类为：**行优先**（步长为 1）、**列优先**（步长为 N）、**固定访问**（循环内地址不变）
- 判断哪种顺序对 Cache 最友好

**分析表格示例：**

| 循环顺序 | A 的访问模式 | B 的访问模式 | C 的访问模式 | Cache 友好度 |
|---------|------------|------------|------------|-----------|
| ijk     | 行优先(+1) | 列优先(+N) | 固定       | 中等      |
| jik     | 行优先(+1) | 列优先(+N) | 列优先(+1) | 较优      |
| kij     | 固定       | 行优先(+1) | 行优先(+1) | 最优      |
| ikj     | 行优先(+K) | 列优先(+N) | 行优先(+1) | 低        |
| jki     | 固定       | 列优先(+1) | 行优先(+N) | 低        |
| kji     | 行优先(+1) | 固定       | 行优先(+N) | 低        |

**关键观察：** 步长为 1 的顺序最友好，因为能够充分利用 Cache 行。`kij` 通常性能最好。

---

### c) Cache 瓶颈分析

**要求：**
- 用 `perf` 性能计数工具测量最快和最慢的两种循环顺序
- 分别记录 L1、L2、L3（LLC）的 miss rate
- 比较差异，找出主要性能瓶颈

**测试命令：**

```bash
# 通用性能统计
perf stat ./program

# 专项 Cache 分析
perf stat -e L1-dcache-load-misses,LLC-load-misses ./program

# 详细计数
perf stat -e cycles,instructions,L1-dcache-loads,L1-dcache-load-misses,L2-dcache-loads,L2-dcache-load-misses,LLC-loads,LLC-load-misses ./program
```

**预期分析：**
- L1 Cache 通常为 32~64KB，为最紧张的层级
- 列优先访问的顺序会导致高 L1 miss rate（>50%）
- 行优先访问的顺序 L1 miss rate 可能低于 10%

---

### d) 数据布局优化：转置 B 矩阵

**要求：**
- 实现两个版本：
  1. **原始版本**：直接按行优先访问 B
  2. **优化版本**：预先将 B 转置为 $B^T$，再进行行优先访问
- 比较性能提升，评估转置开销是否值得

**实现思路：**

```c
// 原始版本（列优先访问 B）
for(int i = 0; i < M; i++)
    for(int j = 0; j < N; j++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];

// 优化版本（转置后行优先访问 B^T）
// 1. 转置 B 为 B_T
for(int k = 0; k < K; k++)
    for(int j = 0; j < N; j++)
        B_T[j][k] = B[k][j];

// 2. 用 B_T 进行计算（B_T[j][k] 对应原 B[k][j]）
for(int i = 0; i < M; i++)
    for(int j = 0; j < N; j++)
        for(int k = 0; k < K; k++)
            C[i][j] += A[i][k] * B_T[j][k];
```

**预期结果：** 即使加上转置开销，总时间通常也会显著减少（可能减少 30~50%），因为转置是一次性的，而矩阵乘法中 B 被访问多次。

---

### e) Cache 关联度与冲突不命中

**要求：**
- 分别在矩阵规模 1001、1024、1025 下测试性能
- 画出性能曲线，观察是否出现陡峭跳变
- 解释为什么 1024（$2^{10}$）这类大小可能性能反而更差

**背景知识：**

CPU Cache 通常采用**组关联** (Set-Associative) 结构：
- L1 Cache：8-way，64KB
- L2 Cache：4-way 或 8-way，256KB~1MB
- L3 Cache：12-way 或 16-way，几MB

当矩阵大小为 $2^n$ 时，多个数据可能映射到同一 Cache 组，导致**冲突不命中** (Conflict Miss)。

**示例分析：**

| 矩阵规模 | 每行字节数 | Cache 映射特性 | 预期性能 |
|---------|----------|-------------|--------|
| 1001     | ~8008    | 无规律      | 基准    |
| 1024     | 8192     | $2^n$ 对齐，易冲突 | **可能下降** |
| 1025     | ~8200    | 无规律      | 基准或略好 |
| 2048     | 16384    | $2^n$ 对齐，冲突更严重 | **大幅下降** |

**实验步骤：**

```c
// 在不同规模下测试
for(int size = 1000; size <= 2050; size++) {
    // 构建大小为 size × size 的矩阵
    // 执行矩阵乘法
    // 记录耗时
    // 输出：size, time_ms
}
```

**预期曲线：** 
- 1001～1023：时间相近
- 1024：**陡峭上升**（冲突不命中激增）
- 1025～2047：时间相对平稳
- 2048：**再次上升**（冲突更严重）

**解决方案（可选扩展）：**
- **Padding**：在矩阵末尾补充列，如 `A[size][size+8]`，破坏 $2^n$ 对齐
- **数据预转置**：改变数据布局
- **动态 Cache 感知分块**：见选题 2

# 深度分析：缓存临界状态与循环优化的本质

## 1. 什么是缓存临界状态

### 定义

**缓存临界状态** = 系统缓存的利用率处于某个"平衡点"或"分界线"，任何微小的操作顺序改变都可能引发显著的性能波动。

### 实验中的三个状态

对分块矩阵乘法（块大小64×64）的36种循环组合进行测试，可以观察到三类不同的缓存行为：

```
✅ 稳定状态（块级ijk/jik）：
   - B块缓存复用频率 = 1（加载后立即被驱逐）
   - 缓存压力：确定性低
   - 性能提升：+42% ~ +50%  （波动 ±3%）
   
✅ 稳定状态（块级kij/ikj）：
   - B块缓存复用频率 = M×N（被所有i,j重复使用）
   - 缓存压力：确定性高
   - 性能提升：-10% ~ -30%  （波动 ±3%）

⚠️ 临界状态（块级jki/kji）：
   - B块缓存复用频率 = M 或 N（中间水平）
   - 缓存压力：处于"边界"，敏感
   - 性能提升：±0% ~ ±5%   （波动 ±7%！）
```

### 为什么会出现波动？

在临界状态下，缓存容量恰好饱和：

```
L2缓存大小 ≈ 256 KB
块级占用 = A[64×64] + B[64×64] + C[64×64] = 96 KB
          → 还剩160 KB空闲

微小的元素级循环改变：
  - i在最内层 → 额外预加载临近A数据 → 占用+8KB
  - j在最内层 → 访问跳跃，无预加载 → 占用+0KB

同样的指令量，缓存压力不同
→ miss率波动
→ 性能波动 ±5%
```

---

## 2. 转置改变的是什么

### 转置不改变循环结构，只改变访问步长

```c
// 块级循环完全不变
for(int ii) for(int jj) for(int kk)
    ↑ 外层三重循环保持原样

    // 内层循环结构也不变
    for(int i) for(int j) for(int k)
        C[i*N+j] += A[i*K+k] * B[k*N+j];     // 不转置
                                    ↑
                                    
        C[i*N+j] += A[i*K+k] * B_T[j*K+k];   // 转置后
                                    ↑
```

### 转置改变的是什么

转置改变的是**元素级循环内的访问步长**：

| 访问对象 | 不转置 | 转置后 | 步长差异 |
|---------|--------|--------|---------|
| **B[k][j]** | 步长=N（列优先）❌ | - | - |
| **B_T[j][k]** | - | 步长=1（行优先）✅ | 改善空间局部性 |

**关键点**：转置改变的是**内层（元素级）的访问效率**，而不是外层（块级）的循环结构。

---

## 3. 为什么外层循环决定一切

### 两层循环的职责完全不同

```
for(块级)           ← 决定 "谁进谁出"（缓存的大进大出）
    for(元素级)     ← 决定 "怎么访问"（块内的访问步长）
        计算
```

### 类比：打扫房间中的遥控器

**块级循环** = **遥控器的位置（哪个房间、哪个抽屉）**
- 放在客厅桌上（块级kij）：每次看电视都能立刻拿到 → **时间局部性高**
- 放在楼上卧室（块级ijk）：每次看电视都要上楼去拿 → **时间局部性低**

**元素级循环** = **按什么顺序翻频道**
- 从1到100 或 从100到1 → 都不改变"上楼拿遥控器"有多麻烦

**转置** = **重新标记遥控器按钮**
- 改变了"按钮的逻辑编号"，但**没有改变遥控器本身的位置**

> 🔑 **所以：无论怎么标记按钮（元素级），都改不了"上楼拿遥控器"有多麻烦（块级），而这麻烦就是性能差异的主要来源**

### 数学角度：Miss基数支配改善收益

```
加速比 = 不转置时间 / 转置后时间
       = (基础时间) / (基础 - 转置改善)

当块级决定miss数少时（kij）：
  比值 ≈ 小数值 / (小数值 + 转置开销) < 1
  → 反而变慢 ❌
  
当块级决定miss数多时（ijk）：
  比值 ≈ 大数值 / (大数值 - 转置改善) > 1
  → 显著加快 ✅
  → 转置的改善被放大了
```

**结论**：
- 转置改变的是**内层（元素级）的访问效率** ✓
- 但内层效率的改善**被外层（块级）的基数所支配** ✓  
- 因此**规律看起来主要由外层决定** ✓

---

## 4. 时间局部性 vs 空间局部性

### 缓存性能的两个维度

| 维度 | 定义 | 改善方式 | 影响量级 |
|------|------|--------|--------|
| **时间局部性** | "同一数据被重复使用" | **改变块级循环** | ±40% ⬅️ **大** |
| **空间局部性** | "相邻数据在同一缓存行" | **改变元素级循环或转置** | ±10% ⬅️ 小 |

### 为什么时间局部性影响更大

```
块级ijk：
  B块缓存miss基数 = N/64 × K/64 ≈ 256 次
  （即使再优化元素级，miss总数也改不了太多）
  
块级kij：
  B块缓存miss基数 = 1 次
  （再优化转置也很难减少多少）
```

**所以转置的空间局部性改善被时间局部性的基数所支配。**

## 5. 规律总结表

### 只看块级循环就能判断转置效果

| 块级循环 | 最内层变量 | B块复用频率 | 转置效果 | 性能提升 |
|---------|----------|----------|--------|---------|
| **ijk** | kk | 1（低）| ✅ 高收益 | **+42%~+50%** |
| **jik** | kk | 1（低）| ✅ 高收益 | **+42%~+50%** |
| **kij** | jj | M×N（高）| ❌ 负收益 | **-25%~-30%** |
| **ikj** | jj | N（高）| ❌ 负收益 | **-10%~-15%** |
| **jki** | ii | M（中）| 🟡 中等 | **±1%~+4%** |
| **kji** | ii | M×N（高）| 🟡 中等 | **±2%~+5%** |

### 快速判断规则

```
if 块级最内层 == kk:
    → 转置有正收益 +45% ✅
    
elif 块级最内层 == ii:
    → 转置无显著收益 ±2% 🟡
    
else: # 最内层 == jj
    → 转置有负收益 -27% ❌
```

## 6. 对实际优化的启示

### 分块矩阵乘法的最优实践

```c
// ✅ 推荐方案：块级ijk + 转置B
for(int ii = 0; ii < M; ii += 64)
    for(int jj = 0; jj < N; jj += 64)
        for(int kk = 0; kk < K; kk += 64)
            // 块内循环任意（元素级无影响）
            for(int i = ii; i < ii+64; i++)
                for(int j = jj; j < jj+64; j++)
                    for(int k = kk; k < kk+64; k++)
                        C[i*N+j] += A[i*K+k] * B_T[j*K+k];

// 性能稳定：+45% 左右
// 原因：块级ijk的低复用被转置的高收益完全补偿
// 元素级顺序无关紧要
```

### 为什么不用其他块级顺序

**块级kij**（看起来最优）实际上最差：
```
理由1：B块已经高度复用 (M×N 次)
       → 转置反而破坏完美的缓存行为
       → 性能下降 -27%
       
理由2：缓存容量已经饱和
       → 转置增加的内存占用会导致驱逐
       → 得不偿失
```

**块级jki/kji**（临界区域）不稳定：
```
理由：缓存压力处于边界
      → 微小的元素级变化会导致 ±5% 的波动
      → 缺乏可预测性
```

## 7. 缓存临界状态的实际意义

### 为什么要了解临界状态

临界状态是系统设计中的**"脆弱区域"**：

1. **可预测性差**：小改动导致大性能差异
2. **可移植性差**：在一个CPU上最优，在另一个CPU上可能变差
3. **调试困难**：性能波动难以追踪根因

### 如何回避临界状态

```
策略1：远离临界状态
       → 选择块级ijk（稳定 +45%）
       → 不要选择块级jki/kji

策略2：精确计算缓存占用
       → 确保块大小 × 矩阵数量 < L2 × 70%
       → 给其他进程预留缓存空间

策略3：使用缓存感知分块
       → 根据具体CPU的L2/L3大小动态调整块大小
       → 避免硬编码64×64
```

## 8. 关键洞察

### 缓存优化的分层认识

| 层级 | 优化对象 | 典型收益 | 难度 |
|------|---------|--------|------|
| **1. 时间局部性** | 块级循环结构 | **40%** | 容易 |
| **2. 空间局部性** | 元素级循环或转置 | **10%** | 中等 |
| **3. 临界区域感知** | 识别不稳定区域 | **避免风险** | 难 |

### 最简洁的设计建议

**对于分块矩阵乘法：**
> 直接使用 `ijk块级 + 转置B`，性能稳定 +45%，无需复杂思考。
